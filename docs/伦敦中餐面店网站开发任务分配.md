# 伦敦中餐面店官网开发任务分配（个人开发者最轻量版）

> 目标：你一个人也能持续维护。优先“简单、稳定、低成本、少运维”。

---

## 1. 先定原则（避免后期复杂化）

1. **能静态就静态**：内容变化不频繁，优先静态页面（Next.js SSG）。
2. **能托管就托管**：尽量用 AWS 托管能力，避免自建服务。
3. **能单仓库就单仓库**：前后端分离会增加维护成本，先做单体前端官网。
4. **能少流程就少流程**：只保留必要流程（开发 -> 合并 -> 自动部署）。
5. **能快速回滚就不怕上线**：每次发布都可一键回退。

---

## 2. 最轻量技术选型（推荐直接照抄）

## 2.1 技术栈
- 前端：`Next.js + TypeScript`
- 样式：`Tailwind CSS`（可选，核心是统一样式）
- 内容：`content/menu.json` + `content/site.json`
- 容器：`Docker`
- 云：`ECR + ECS Fargate + ALB + Route53 + ACM`
- CDN：`CloudFront`（只放图片/静态资源）
- 日志：`CloudWatch Logs`

## 2.2 为什么这是最轻量
- 不引入数据库（首版）
- 不引入 CMS（首版）
- 不做微服务
- 不做复杂 IaC（首版可控制台搭建，后续再 Terraform）

---

## 3. 网站最小功能（MVP）

必须页面（6 个）：
1. Home
2. Menu
3. About
4. Contact
5. Reserve（跳 OpenTable / 电话 / WhatsApp）
6. Privacy（合规页）

必须信息：
- 门店地址（伦敦）
- 营业时间
- 电话/邮箱
- 菜单价格
- 过敏原提示

> 注意：个人开发阶段先别做会员系统、在线点餐、支付。

---

## 4. 单人开发流程（最短路径）

## 4.1 分支策略（极简）
- `main`：线上生产
- `dev`：开发测试

流程：
1. 在 `dev` 开发
2. 本地自测通过
3. `dev` 合并到 `main`
4. GitHub Actions 自动部署生产

## 4.2 本地开发 SOP
1. 拉代码
2. 修改页面或 `content/*.json`
3. 本地运行 + 自测
4. 提交 Git
5. 合并并自动部署

---

## 5. AWS 部署流程（最轻量）

## 5.1 一次性搭建（首周只做一次）
1. 创建 ECR 仓库（存镜像）
2. 创建 ECS Cluster（Fargate）
3. 创建 Task Definition（容器端口 3000）
4. 创建 ECS Service（最小 1 个任务）
5. 创建 ALB（80/443）
6. ACM 申请证书（根域 + www）
7. Route53 配置域名到 ALB

## 5.2 每次发布（自动）
- push `main` -> GitHub Actions：
  1) build 镜像
  2) push ECR
  3) 更新 ECS Service

## 5.3 回滚（手动但简单）
- ECS 里切回上一个 Task Definition 版本。
- 目标：5~10 分钟恢复。

---

## 6. 域名连接服务器（你最关心）

## 6.1 必要对象
- 域名（如 `yournoodle.co.uk`）
- Route53 Hosted Zone
- ACM 证书（`yournoodle.co.uk` + `www.yournoodle.co.uk`）
- ALB（绑定证书）

## 6.2 最简步骤
1. Route53 建 Hosted Zone
2. ACM 申请证书并 DNS 验证
3. ALB 的 443 监听器绑定证书
4. Route53 配置 `A Alias` 到 ALB
5. ALB 80 -> 443 重定向

## 6.3 验证清单
- `https://yournoodle.co.uk` 可打开
- `http://` 会自动跳 HTTPS
- 页面加载无证书警告

---

## 7. 内容更新流程（非程序员也可执行）

## 7.1 最轻量方案：改 JSON
- 菜单在 `content/menu.json`
- 店铺信息在 `content/site.json`

更新步骤：
1. 改 JSON
2. 提交到 `dev`
3. 检查预览
4. 合并 `main`
5. 自动上线

## 7.2 什么时候再上 CMS
满足任一条件再升级：
- 每周改内容 > 3 次
- 需要多人编辑
- 需要定时发布活动
